Topic Context

The build. FDE-os is a new, empty project (~/Documents/Projects/FDE-os) with three objectives: (1) a personalizable, multi-mastery-level course that upskills a person toward Forward-Deployed-Engineer roles, validated by the question "given a job description, does this course train the person well enough to land that job?"; (2) FDE tooling — skills, plugins, dynamic workflows, hooks — for AI coding agents (Claude, Codex, Hermes, OpenClaw); and (3) a field-practice feedback flywheel — real engineers run the Objective-2 tooling inside real customer engagements, the tooling auto-journals their real use cases, struggles, and problem-solving, and that field signal feeds back to improve Objectives 1 and 2.

Three grounding sources. ai-engineering-from-scratch is the blueprint for how to build a curriculum-as-repo: a phase/lesson tree where every lesson has docs/en.md + runnable code/ + a 6-question quiz.json + an outputs/ artifact; a non-negotiable "Build It / Use It" spine (first principles, then the production library); docs as source-of-truth with a generated website (site/build.js); CI invariant tooling (audit_lessons.py, build_catalog.py, scaffold-lesson.sh, install_skills.py); and seed personalization skills (find-your-level, check-understanding).

The domain content comes from the Interview Kickstart FDE program (a 23-week live cohort): an AI-Engineering spine (agents, RAG, multi-agent, MCP/A2A, observability/eval/safety, fine-tuning) plus the distinctly-FDE craft — customer discovery, SoW & pricing, RBAC/“plumbing”, governance/compliance, evals-as-deploy-gate, incident handover — taught across three clouds (OpenAI+Azure, Google ADK+Vertex, Claude+Bedrock), ending in interview prep.

The target outcome is exemplified by the Google Cloud Forward Deployed Engineer IV, GenAI JD: an embedded builder who ships bespoke agentic systems inside the customer’s environment, removing production blockers (integration, data-readiness, state-management); min quals 3 yrs Python + ML, applied AI around pretrained models (prompt/fine-tune/RAG/tool-orchestration), cloud deploy; preferred multi-agent frameworks (LangGraph/CrewAI/ADK), ReAct/self-reflection/hierarchical delegation, LLM-native metrics (tokens/sec, cost/request), tracing. A 4-level ladder (I→IV→Staff) exists.

Central tensions. (a) IK is a live cohort; the ask is self-serve and personalizable — so the "customer" and "instructor" must be supplied by agents. (b) ai-engineering-from-scratch teaches AI from scratch, but the FDE role needs the top of that stack plus customer-facing craft that pure-eng curricula omit — and that craft has no "raw math" to derive from. (c) The JD-fit ask is a generalization of find-your-level: map a JD (not a person) to a required-competency set, diff it, gate readiness.

Objective 3 — the feedback flywheel. This is a designed-in loop, not the "wait for alumni" idea cut in the first pass: the journaling rides on the Objective-2 tooling practitioners already run, so field signal exists the moment anyone uses a tool. It literally embodies the Google JD ("act as a critical feedback loop, transforming real-world field insights into the product roadmap") — FDE-os teaches the feedback-loop competency by being one. Privacy is load-bearing: real engagements carry NDAs, PII, and SOC2/HIPAA/FedRAMP/GDPR constraints, so the journal must capture the engineering struggle (what broke, which pattern fixed it) without leaking customer specifics — redaction at capture time, reusing the Presidio competency the course already teaches. The captured corpus is the highest-fidelity fuel for the simulator (I5), the eval-gate (I4), and the self-authoring loop (I6): fictional scenarios decay; real ones don't.

The cloud reference — github.com/wjlgatech/data-architecture. An open-source Claude plugin = "Data Architect School," a 5-day curriculum delivered AS skills (skills / knowledge-base / schemas / templates / examples, JSON-validated, MIT). It serves FDE-os twice: as the cloud/data content the JD demands ("data readiness" is a named production blocker) — Medallion Lakehouse, Data Vault 2.0 / Star / 3NF, Data Mesh/Fabric/Lambda, Azure-primary (Databricks, Synapse, Delta Lake, ADF, Event Hub, MLflow) with AWS/GCP refs, governance (MDM, HIPAA/GDPR, zero-trust, break-glass audit), and the NewLife Hospital case (200M+ FHIR records, multi-jurisdiction compliance, $127M NPV roadmap) — and as a second delivery-model reference: a curriculum shaped natively as a schema-validated plugin-of-skills, a packaging that FDE-os tooling (Objective 2) can adopt directly.

I1

JD-Compiler: a job description is the input format, not a fixed curriculum

Axis JD-fit & personalizationConfidence 85%Complexity High

Generalize find-your-level (10 questions → starting phase) into parse-jd: paste any FDE job description, the engine extracts a required-competency vector, diffs it against the learner’s proven state, and emits the minimal lesson path plus a hard "ready for THIS job?" gate that can refuse — "Not ready for FDE-IV: 0 of 3 multi-agent-pattern proofs (ReAct, self-reflection, hierarchical delegation)." Two refinements the frames converged on: crux-first ranking (name the single gating competency, not a flat 12-item gap list) and a JD-matched evidence dossier (reuse the site/build.js generator to render the learner’s outputs/ against a specific posting — the artifact that actually lands the job).

Basis
direct: grounding — "The JD-fit ask = a generalization of find-your-level: map a JD to a required-competency set, diff against the learner, and emit a personalized path + readiness gate." The Google JD supplies the competency vocabulary (LangGraph/CrewAI/ADK, ReAct/self-reflection/hierarchical delegation); the I→Staff ladder makes "ready for level N" meaningful.
Rationale
This is the project’s literal validation question made executable. Most courses can only say "you’re making progress"; a gate that can say no, with a named crux, is the entire reason FDE-os exists — and it differentiates a function (JD → path → proof) from yet another course.
Downsides
Competency extraction from prose JDs is fuzzy and needs a stable taxonomy (see I2). A wrong "not-ready" verdict erodes trust fast. The "Residency-Match" extension (rank real open jobs) presumes a job/learner marketplace that doesn’t exist yet — defer it.
I2

One mastery scale: collapse CEFR ↔ FDE ladder ↔ JD level into a single number

Axis Curriculum content & structureConfidence 88%Complexity Low

The user named three things separately — "mastery levels," the FDE career ladder (I/II/III/IV/Staff), and JD seniority — but they are the same latent variable. Define one provider-agnostic competency-depth scale, expressed as CEFR-style can-do descriptors: L-II — "can build a RAG agent against a clean dataset under supervision"; L-IV — "can independently scope, price, and ship a multi-tenant agent into a regulated customer cloud with a handover runbook." Every lesson tags the depth it reaches; the same RAG lesson reads differently at L-II vs L-IV. The readiness gate (I1) becomes a numeric comparison instead of a fuzzy mapping.

Basis
direct: grounding — "mastery levels map naturally onto the FDE ladder (I→IV→Staff) and to JD levels." external: CEFR (the global language-fluency framework) proved that outcome-described, provider-agnostic "can-do" levels become a shared vocabulary employers trust without knowing the school.
Rationale
This is the cheapest, most foundational primitive — the enabling substrate for I1’s gate, I3’s per-level lesson reads, and I5’s grading. Choosing the scale early makes every downstream feature a comparison on one axis instead of N pairwise mappings.
Downsides
Authoring credible, discriminating can-do descriptors per competency is real editorial work; vague descriptors collapse the scale’s value. Calibrating "L-IV" to what hiring managers actually mean requires JD-corpus evidence to avoid drift.
I3

FDE-Craft as first-class content — extend Build-It/Use-It with a “Deploy-It” rung

Axis Curriculum content & structureConfidence 84%Complexity Medium

The reference repo’s spine is "Build It (from scratch) / Use It (production library)." FDE work needs a third rung the pure-eng repo never required: Ship It Into a Customer Environment — the same artifact deployed under per-tenant RBAC, identity (Entra/workload identity), audit logging, a kill-switch, and a simulated SOC2/HIPAA constraint. And the distinctly-FDE craft (discovery, SoW & pricing, governance, handover) is taught "from scratch" = from first principles, not from math: the learner drafts a one-page problem brief, a risk-buffered SoW, a DECISIONS.md, an RCA — each shipping as a reusable outputs/ artifact, satisfying the repo invariant. Folds in the "pattern-core + stack-adapter" idea: factor each lesson into a cloud-neutral core (ReAct, planner-executor-critic, RBAC-at-tool-boundary) plus thin adapters, so Use-It runs on the learner’s target cloud (GCP→Vertex, Azure→Foundry) without 3× lesson sprawl.

Basis
direct: grounding — "FDE craft is the differentiator and the hardest to teach from scratch"; the IK FDE Spine enumerates the exact deliverables (problem brief, DECISIONS.md, RACI, 4-milestone SoW, RBAC at tool boundary, kill-switches, autonomy ladder). The JD centers the whole role on "production reality inside the customer’s env," which Build-It/Use-It alone omits.
Rationale
This is the moat. Every from-scratch AI course teaches RAG and agents; almost none teach the field craft that defines the FDE role. Extending the repo’s sacred invariant (rather than discarding it) keeps a proven structure while adding the differentiator.
Downsides
Craft has no objective "correct answer" the way code does — grading discovery/SoW quality needs the simulator (I5) and rubrics. "Deploy-It" against real RBAC/compliance is heavy infra to simulate convincingly; risk of teaching cargo-cult governance if the simulation is shallow.
I4

Eval-as-Gate proof-of-mastery — the rubric IS a production deploy-gate

Axis Assessment & proof-of-masteryConfidence 86%Complexity Medium

The FDE curriculum teaches "eval-as-CI-deploy-gate" as content; turn that same machinery inward. A capstone "passes" mastery only when the learner’s agent clears an automated eval suite (DeepEval/LangSmith rubrics, a cost-per-request ceiling, guardrail/PII checks, an autonomy-tier assertion) — exactly the gate a real FDE must build for a customer. Proof-of-mastery becomes a green eval transcript, not a quiz score or instructor sign-off, removing subjective grading for the production phases. An employer could re-run the same gate on the candidate’s repo and trust the result.

Basis
direct: grounding — IK Spine teaches observability/eval/safety (LangSmith, DeepEval, Guardrails AI) and the FDE Spine teaches "eval-as-CI-deploy-gate"; the reference repo already runs lesson_run.py / audit_lessons.py as CI-blocking invariants, so "the test is the gate" is in the repo’s DNA.
Rationale
The JD explicitly wants "not just demos." A 6-question quiz proves recall; an eval-gate proves the thing an FDE is hired for — shipping an agent that survives production checks. One eval-harness investment serves teaching, grading, and portfolio proof at once.
Downsides
Writing fair, non-gameable eval suites per capstone is hard; learners may overfit to the gate. Eval infra (LLM-as-judge, golden sets) has its own cost and flakiness. Best paired with I5 so customer-craft (not just code) is also gated.
I5

The Customer Simulator — a standardized-customer agent that fights back

Axis Assessment & proof-of-masteryConfidence 80%Complexity High

The hardest-to-replicate part of FDE training is the customer. Build an agent that plays a standardized stakeholder — the evasive VP, the security-paranoid CISO, the budget-cutting procurement lead — each with a hidden problem, a fixed budget, a real cloud/compliance posture, and a RACI map. The learner runs the 90-minute discovery workshop, a procurement drill, and an injected incident fire-drill; a rubric agent scores the artifacts they produce. Track it as accumulating, per-scenario "sim hours" mapped to the I2 ladder, with an after-action review on each capstone run.

Basis
external: the medical OSCE (standardized-patient stations, Harden 1975) and US Army after-action review (TC 25-20) are the proven structures for certifying interpersonal/under-failure competence that written exams can’t. reasoned: IK delivers this craft in a live cohort where humans play the customer; a self-serve course literally cannot offer discovery/procurement practice without an agent playing that role — and the project already commits to agent runtimes.
Rationale
It unlocks the entire FDE Spine (the differentiator) for self-serve delivery and is the self-serve answer to the cohort program’s strongest feature.
Downsides
A persona agent that’s too easy is worthless; calibrating realistic difficulty and fair scoring is the core risk. Highest build complexity of the survivors — sequence it after I2/I3 give it a scale and artifacts to grade.
I6

Self-authoring pipeline — PDF/JD in, audited lessons out

Axis Authoring, generation & maintenanceConfidence 80%Complexity Medium

Hand-authoring at curriculum scale is the maintenance bottleneck. Build an authoring agent that ingests a JD or a domain PDF (like the IK FDE PDF), extracts competencies, checks them against the catalog, and either maps to existing lessons or scaffolds new ones via scaffold-lesson.sh — pre-filled with the Build-It/Use-It skeleton, a 6-question quiz, and an artifact target. Crucially, audit_lessons.py (already CI-blocking) becomes the accept gate that rejects any malformed generation. The human shifts from author to reviewer-of-machine-output; run the same JD parser over a corpus to surface coverage gaps that auto-file as authoring tickets.

Basis
direct: grounding axis 5 explicitly names "agent-generated lessons from JDs/PDFs," and the repo ships scaffold-lesson.sh + build_catalog.py + audit_lessons.py as the exact rails such a generator would target. reasoned: a rigid, machine-checkable lesson schema + a CI invariant auditor is precisely the precondition that makes LLM-generated content safe to ship — the auditor catches malformed generations, so generation cost drops without quality dropping.
Rationale
The JD landscape moves fast (LangGraph/CrewAI/ADK are all recent). A hand-maintained curriculum decays; closing the loop "new JD → catalog gap → generated stub" makes the curriculum self-extending and keeps the JD-fit engine (I1) honest as roles drift.
Downsides
The auditor checks structure, not pedagogical truth — generated lessons can pass CI while being subtly wrong, so human review stays load-bearing. Garbage-PDF-in risks garbage-lessons-out. Don’t over-build the corpus-scraping "coverage oracle" before there’s a curriculum to measure.
I7

Write-once, run-on-four-runtimes — the FDE artifact compiler

Axis FDE agent tooling layerConfidence 70%Complexity High

This is the second half of the ask. Author each FDE-craft artifact once in a runtime-neutral spec (a "scope-a-customer" skill, an "RBAC-tool-boundary" hook, a "cost-shape estimator" workflow, an "incident-RCA" runbook) and compile it to Claude skill / Codex / Hermes / OpenClaw formats — the way site/build.js compiles markdown to data.js. Ship each with a portability conformance test: the artifact’s proof-of-value is a measured before/after pass-rate delta on a held-out FDE task across at least two of the four runtimes (the "agent-as-student" acceptance test). An FDE never controls which agent stack the customer runs, so write-once/run-anywhere is exactly the plumbing the role rewards.

Basis
direct: grounding — install_skills.py "copies course outputs into a target agent dir as skills," and "cross-agent portability (4 different agent runtimes) is a real design axis"; the docs→site compile is the existing precedent. reasoned: flipping the "student" to the agent gives every artifact an objective, automatable acceptance test instead of self-declared value.
Rationale
Directly serves the JD’s "ship bespoke agentic solutions inside the customer’s env" — the learner’s reward for a lesson is a working tool installable into whatever agent the customer runs, with proof it helps.
Downsides
Lowest-confidence survivor: the four runtimes’ skill/hook formats differ enough that a clean neutral IR is non-trivial, and today’s precedent (install_skills.py) only copies, it doesn’t compile. Premature to design the IR before a single artifact exists — build one artifact for one runtime first, then generalize.
I8

Field-Journal event log — the flywheel hub (Objective 3)

Axis Field practice & feedback flywheelConfidence 85%Complexity Medium

The core of Objective 3 is a single asset: a structured, redacted engagement event log. As an engineer runs the Objective-2 tooling in a real engagement, each tool call, struggle, and recovery is emitted as one typed record — {struggle_type, pattern_invoked, tool_called, error_class, recovery_move, time_to_fix} — with all customer entities replaced by typed placeholders. One asset, four consumers: it mints I4 eval cases, I5 simulator scenarios, I6 curriculum-gap tickets, and I7 artifact-fix issues. Build the schema once; every downstream survivor reads from it, so each field hour makes all four sharper.

Basis
direct: grounding — "structured, machine-usable signal: pain points → curriculum gap tickets (feeds I6) and tool failure reports → artifact fixes (feeds I7)"; "highest-fidelity source of new Customer-Simulator (I5) scenarios and new eval-gate (I4) test cases." Reuses the repo’s outputs/-index aggregation pattern.
Rationale
This is the leverage point of Objective 3. Without one canonical schema, each survivor grows its own ad-hoc ingest and the data fragments; with it, the marginal cost of a new consumer is a query, not a pipeline.
Downsides
Schema design is a one-way door — get the typed fields wrong and every consumer inherits the gap. Cold-start: the log is empty until people use the tools in the field (mitigated by I11’s incentive and I12’s case-study seed data).
I9

Redact-at-capture, anchored to the diff — born-clean journaling

Axis FDE agent tooling layerConfidence 84%Complexity Medium

Make the journal a PostToolUse/Stop hook that never writes raw transcript. Every captured turn passes through a local Presidio + abstraction pass before a byte touches disk: customer names → <CUSTOMER>, schemas → shape-only, secrets dropped, prose rewritten to "the engineering struggle." Better still, anchor capture to the engineer’s git activity rather than LLM chatter: record the delta from a failing state to a passing one (failing test → commits → passing test, plus the tool calls in that window). A fix-shaped diff is intrinsically low-PII, structured, and maps 1:1 onto an I4 eval case or I5 scenario.

Basis
direct: grounding — "redaction/abstraction at capture time is mandatory, not optional … reuse Presidio on the journal itself"; the base course already teaches Presidio PII redaction. reasoned: transcripts are high-entropy and PII-dense; git diffs are low-entropy and structured, and git is already running in every engagement, so anchoring there adds zero new capture surface.
Rationale
This is the precondition for the whole flywheel: if raw customer data physically cannot be written, a customer’s security team can plausibly approve the capture path — without that sign-off, field journaling never happens. It also dogfoods the exact redaction competency the course sells.
Downsides
Redaction is never perfect — a residual-leak audit and a kill-switch are mandatory, and over-redaction can strip the signal that made the struggle teachable. Git-anchoring misses non-code struggles (a botched discovery call), so it complements, not replaces, the I5 simulator capture.
I10

Pain-to-ticket compiler + freshness decay — the loop’s back half

Axis Authoring / field flywheelConfidence 82%Complexity Medium

The flywheel’s failure mode is that signal pools and nobody acts. Add the back half: a scheduled job clusters abstracted struggles across all engineers and emits two ranked queues — curriculum gap tickets (struggle types with no matching lesson → feed I6, prioritized by how many distinct customers hit them) and tool-failure reports (skills that errored or were abandoned → feed I7). The same provenance data drives a decay score: tag every I4 eval case and I5 scenario as fictional vs field-derived with a last-seen-in-field timestamp; down-rank stale fictional cases and promote recurring real ones. The corpus self-curates toward what engineers hit this quarter.

Basis
direct: grounding — "pain points → curriculum gap tickets (feeds I6)"; "fictional scenarios decay; real ones don’t." Reuses the repo’s audit_lessons.py/build_catalog.py CI-invariant aggregation pattern as the harness.
Rationale
Without this, capture is a write-only log. Frequency-ranked tickets turn "practice → curriculum improvement" from aspiration into a mechanical roadmap the field votes on; decay scoring adds the forgetting that keeps I4/I5 from testing problems nobody faces anymore — a self-curating eval set competitors with static banks can’t match.
Downsides
Clustering quality is the whole game — bad clustering produces noisy or duplicate tickets that erode trust. Frequency bias can bury rare-but-critical struggles (a single catastrophic compliance failure outranks 12 trivial ones); needs a severity dimension, not just count.
I11

Engineer-owned field portfolio — the incentive engine

Axis Field practice & feedback flywheelConfidence 83%Complexity Low

The flywheel only spins if engineers leave journaling on during real, NDA-bound work — and they won’t if it feels like surveillance-for-the-platform. Invert ownership: the journal is the engineer’s private asset first, always building their personal retro corpus and a portfolio/evidence stream locally (auto-generating STAR-format interview stories, mapped onto the I2 mastery scale and the FDE I→IV ladder). Contributing to the shared improvement corpus is a separate, explicit, per-entry opt-in that re-runs a stricter abstraction pass. Personal value funds collective compounding; sharing is a bonus, not the price of entry.

Basis
direct: grounding — "the journal is also a per-engineer asset: a portfolio/evidence stream and a personal retro corpus"; base grounding’s STAR career track and the FDE ladder as mastery levels. reasoned: flywheels die at cold-start when the data source is extractive; making the default benefit personal aligns the incentive with the person doing the work, so adoption survives a customer’s legal team.
Rationale
This is what keeps I8 fed. It also closes the loop back to Objective 1: the same telemetry that improves the course is, sliced per-person, the JD-matched evidence dossier (I1) that lands the next job.
Downsides
Opt-in sharing means the shared corpus may stay sparse if few contribute — needs a genuinely compelling personal payoff and possibly aggregate-only contribution modes. Local-first storage shifts some security burden onto the engineer’s own machine.
I12

Adopt data-architecture as the Data & Cloud track + packaging spec

Axis Content / toolingConfidence 85%Complexity Low–Medium

The JD names "data readiness" as a top production blocker, and a finished, MIT-licensed Claude plugin already teaches it. Vendor wjlgatech/data-architecture as FDE-os’s "Data & Cloud Foundations" track — Medallion Lakehouse, DV2.0/Star/3NF, Data Mesh, Azure-primary stack, governance — via install_skills.py-style wiring rather than re-authoring five days of content. Wrap each imported skill with a thin Build-It/Use-It + Deploy-It (I3) shim. Two compounding bonuses: its NewLife Hospital case (200M+ FHIR records, multi-jurisdiction compliance, NPV roadmap) is a turnkey I5 simulator engagement and cold-start seed for the flywheel; and its schema-validated plugin-of-skills shape becomes the canonical package format the I7 cross-agent compiler targets and emits.

Basis
direct: grounding — data-architecture is a "ready-made Data & Cloud Foundations track to plug into the FDE curriculum rather than re-author"; it is "built natively as a Claude plugin of skills … FDE-os tooling (Objective 2) could adopt/extend this plugin-of-skills packaging directly"; the JD names data readiness; install_skills.py already copies outputs into agent dirs.
Rationale
Highest-leverage content move available: an entire validated track for the cost of integration glue, and its case studies are pre-built realistic scenarios — exactly the non-fictional fuel I4/I5 want before a single real engagement happens, seeding the flywheel’s cold start. Standardizing on its JSON-schema bundles gives I7 a typed surface to compile and lint.
Downsides
It is Azure-primary; a GCP-targeted FDE candidate (the Google JD) needs GCP coverage the plugin only references — pairs with I3’s pattern-core/stack-adapter factoring. Vendoring a third-party repo adds an upstream dependency to track, and its skill shape may need adaptation to match FDE-os’s lesson contract.
